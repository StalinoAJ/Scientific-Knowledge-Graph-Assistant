\section*{ABSTRACT}
\noindent{\rule{\textwidth}{1.5pt}}
The exponential growth of scientific literature makes it increasingly challenging for researchers to stay abruptly informed about state-of-the-art advancements, relevant methodologies, and collaborating authors. Traditional keyword-based search and recent Retrieval-Augmented Generation (RAG) paradigms primarily rely on vector similarity matching. While this captures semantic closeness, it inherently lacks the ability to comprehend explicit structural relationships between distinct research components such as papers, authors, datasets, and proposed computational methods.

To address this critical gap, this dissertation presents the \textbf{Scientific Knowledge Graph Assistant}, an AI-powered, open-source research platform that seamlessly integrates Knowledge Graphs (KG) with Large Language Models (LLMs) to perform Graph Retrieval-Augmented Generation (GraphRAG). The developed system locally ingests research papers from diverse sources including arXiv, PubMed, and Semantic Scholar, extracting structural metadata alongside deep semantic content. By utilizing a local Llama 3.2 model orchestrated via Ollama, the platform parses raw text to construct a deeply normalized graph database in Neo4j comprising \textit{Paper}, \textit{Author}, \textit{Method}, \textit{Dataset}, and \textit{Institution} entities. 

The application architecture includes a highly scalable Python FastAPI backend processing complex multi-hop Cypher queries and a reactive frontend constructed with React, TypeScript, and Cytoscape.js for interactive data visualization. Not only does this system operate at zero API cost by leveraging local containerized inferencing, but it also provides a premium, visually-rich graphical interface allowing researchers to dynamically explore citation networks, investigate trend analytics over time, and submit natural language questions answered securely by grounded, contextually-aware GraphRAG algorithms. The results indicate a substantial improvement in the factual accuracy and structural contextualization of generated answers compared to baseline vector-only approaches, proving the vast potential of local graph-enhanced LLMs in an academic setting.

\chapter{INTRODUCTION}
\noindent{\rule{\textwidth}{1.5pt}}
\pagenumbering{arabic}
\setcounter{page}{1}

The advent of Artificial Intelligence and Large Language Models (LLMs) has vastly transformed the way we access, synthesize, and interact with large corpuses of text. Within the scientific community, the volume of published literature continues to expand at an unprecedented rate, creating a cognitive overload for scientists, students, and research institutions striving to maintain state-of-the-art awareness. The "Scientific Knowledge Graph Assistant" aims to address this data deluge by acting as a locally-hosted, zero-cost, privacy-respecting AI assistant configured to autonomously ingest, map, and query intricate networks of scientific literature.

\section{Objective}
The primary objective of this thesis is to architect, develop, and validate the Scientific Knowledge Graph Assistant. Specifically, the objectives entail:
\begin{itemize}
    \item Creating an automated data ingestion pipeline capable of scraping and normalizing extensive volumes of research papers from reputable sources (arXiv, PubMed, Semantic Scholar).
    \item Designing an optimized Graph Database schema within Neo4j to structurally represent papers, authors, citations, datasets, and methods as disparate entities interconnected by relational semantics.
    \item Implementing a hybrid local-LLM engine utilizing Llama 3.2 via Ollama to perform cost-free embedding generations and complex text extractions, removing dependencies on paid cloud APIs.
    \item Engineering a functional GraphRAG (Graph Retrieval-Augmented Generation) pipeline that marries semantic vector search with explicit relational path traversals to mitigate LLM hallucinations and provide deeply contextualized scientific answers.
    \item Developing an interactive, analytical front-end interface utilizing React and Cytoscape.js to let users visually manipulate, explore, and query the resulting knowledge network.
\end{itemize}

\section{Motivation}
The genesis of this research stems from the inherent limitations of modern search engines and standard RAG deployments. While vector-database RAG systems successfully identify documents that semantically overlap with a query, they often blindly mash these texts together without understanding their structural significance. For instance, answering complex multi-hop queries such as "Which deep learning methods developed by researchers at MIT improved protein folding accuracy after 2020?" requires connecting geographical data (MIT) to temporal data (post-2020), relational data (Authors at MIT), and semantic methodological descriptions (Deep learning for protein folding). 

Furthermore, existing proprietary solutions like OpenAI APIs or commercial knowledge-graph products represent a significant financial overhead and potential privacy risk for independent researchers or smaller institutions. By containerizing open-source LLMs alongside high-performance graph databases, we democratize access to advanced scientific data synthesis tools.

\section{Scope}
The scope of this project is confined to the complete software lifecycle design—from requirement capture to implementation and empirical evaluation—of the Scientific Knowledge Graph Assistant. Functionally, the system scope includes:
\begin{itemize}
    \item **Backend Ecosystem**: Developed using Python's FastAPI, serving REST endpoints for ingestion, semantic parsing, graphing, querying, and statistical analysis.
    \item **Graph AI Integration**: Managing interactions with Neo4j Community Edition and local instance deployments of Ollama.
    \item **Frontend Ecosystem**: A TypeScript/React Single Page Application focusing strictly on visualizing force-directed graph interactions, rendering dark-themed UI components, and displaying dynamic trends.
    \item **Deployment Context**: Ensuring reproducible deployments across varying operating systems exclusively via Docker and Docker Compose orchestration mechanisms.
\end{itemize}

\chapter{REQUIREMENT ANALYSIS AND SPECIFICATION}
\noindent{\rule{\textwidth}{1.5pt}}

The requirement analysis phase marks the initial step in the software engineering process, aiming to deeply comprehend the user's needs, define the existing landscape's deficiencies, and outline exact system requirements to govern the development lifecycle of the Scientific Knowledge Graph Assistant.

\section{Requirement Analysis}

\subsection{Existing System}
In the current ecosystem, researchers rely heavily on standard academic search engines (e.g., Google Scholar, IEEE Xplore, or base PubMed). 
\paragraph{Limitations of the Existing System:}
\begin{itemize}
    \item \textbf{Keyword Dependency}: They are mostly limited to exact or partial keyword matches, frequently returning irrelevant noise.
    \item \textbf{Static Results}: They produce static lists of links. They cannot synthesize answers, compare methodologies, or deduce relational insights across multiple papers simultaneously.
    \item \textbf{High Cognitive Load}: The user must manually read, parse, and verify each retrieved article linking authors or shared datasets manually via memory or separate notes.
    \item \textbf{Basic RAG Deficits}: Recent basic RAG systems use semantic vectors but lose author-affiliation or temporal context entirely because documents are chunked into isolated arrays of text.
\end{itemize}

\subsection{Proposed System}
The proposed Scientific Knowledge Graph Assistant overcomes these boundaries by structuring the data immediately upon ingestion. By extracting entities (Authors, Organizations, Methods) and saving them as explicit graph nodes in Neo4j, while simultaneously embedding the text into vector spaces, the system establishes a highly interconnected hybrid map. 
\paragraph{Advantages of the Proposed System:}
\begin{itemize}
    \item \textbf{Contextual Multi-Hop Reasoning}: It traversers author relationships and citations explicitly, preventing logical disconnections in generated answers.
    \item \textbf{Zero-Cost Execution}: Relying on Llama 3.2 run via Ollama mitigates all SaaS API expenditures.
    \item \textbf{Visual Discoverability}: Users can interactively click and expand nodes to discover new papers visually, something standard search interfaces completely lack.
    \item \textbf{Grounded Generation}: LLM responses include exact graph-verified citations, effectively eliminating hallucinated references.
\end{itemize}

\section{Requirement Specifications}

\subsection{Functional Requirement}
Functional requirements represent the operational features the system must exhibit.
\begin{enumerate}
    \item \textbf{Multi-Source Ingestion}: The system must query and download metadata from arXiv, PubMed, and Semantic Scholar using user-defined scientific keywords.
    \item \textbf{Data Extraction and Normalization}: The backend must employ the local LLM to parse abstracts and identify entities (Methods, Datasets) mapping them dynamically to the graph.
    \item \textbf{Natural Language Query Processing}: The system must accept natural language queries and utilize an LLM intent-parser to translate requests into hybrid Cypher-Vector operations.
    \item \textbf{Graph Traversal and Answer Generation}: The system must fetch neighbors of relevant nodes, inject this topological context into the LLM prompt, and return a streaming response.
    \item \textbf{Interactive UI Rendering}: The frontend must fetch Cytoscape-compatible JSON data representing the network and render it without UI blocking.
\end{enumerate}

\subsection{Non Functional Requirement}
Non-functional requirements describe system attributes such as performance, usability, and reliability.
\begin{enumerate}
    \item \textbf{Performance}: The local LLM inference should stream the first token back to the user within 3-5 seconds depending on hardware limits. 
    \item \textbf{Scalability}: The Neo4j database must handle at least 100,000 nodes efficiently without severe query degradation by utilizing distinct graph indexing.
    \item \textbf{Usability}: The client interface must utilize modern aesthetics, a dark theme for extended reading comfort, and responsive web design suitable for desktop environments.
    \item \textbf{Security \& Privacy}: All processing must occur strictly on the host machine. No scientific queries or datasets shall be transmitted to external cloud providers.
\end{enumerate}

\section{Feasibility Study}
A comprehensive feasibility study guarantees that the implementation of the assistant is viable across various metrics.

\subsection{Operational Feasibility}
The system is operationally viable because academic researchers implicitly understand the domain of citations, authors, and methodologies. The transition from searching lists to exploring visual graphs and asking natural language questions is highly intuitive and matches modern interaction paradigms (like ChatGPT), requiring virtually zero training for the end-user.

\subsection{Technical Feasibility}
Technically, achieving a fully local GraphRAG system was nearly impossible historically due to the immense hardware requirements of LLMs. However, with the release of highly optimized quantization techniques (like GGUF) and frameworks like Ollama, running a powerful model like Llama 3.2 on consumer hardware (16GB RAM) is now entirely achievable. Furthermore, Docker simplifies the complex installation of Neo4j, Python, and Node.js into a single command wrapper.

\subsection{Economic Feasibility}
The project boasts exceptional economic feasibility. As opposed to traditional SaaS solutions which charge per-token for embedding and inference (resulting in thousands of dollars in costs for large research arrays), this architecture leverages entirely open-source, locally executable components. The software cost is conclusively zero.

\section{Tools}
A highly curated technology stack was chosen to maximize open-source viability, speed, and mathematical capability.

\subsection{Python \& FastAPI}
Python acts as the lingua franca of Artificial Intelligence. Its vast ecosystem simplifies integrating LLM processing and manipulating network topologies. Specifically, \textbf{FastAPI} was selected over Django or Flask due to its asynchronous execution architecture, automatic OpenAPI (Swagger) documentation generation, and exceptional processing speed crucial for handling heavy data extraction workloads concurrently.

\subsection{Neo4j Graph Database}
\textbf{Neo4j Community Edition} serves as the core persistence layer. Unlike traditional relational SQL databases where massive JOIN operations become exponentially slower, Neo4j stores relationships as first-class citizens (Index-Free Adjacency). This mechanism is an absolute necessity for querying multi-hop citations and discovering graph triangles in real time. Neo4j also features an integrated Vector Index capable of storing the numeric embeddings generated by the LLM.

\subsection{Ollama \& Llama 3.2}
\textbf{Ollama} provides a lightweight, highly optimized runtime for executing large language models locally utilizing quantized model weights (GGUF formats). \textbf{Llama 3.2} was selected as the underlying intelligence due to its superior reasoning benchmarks relative to its extremely small parameter size, making it feasible to perform complex entity-extraction and natural language generation on consumer-grade hardware without degrading the graph building process.

\subsection{React \& TypeScript}
The frontend logic relies on \textbf{React}, allowing for modular, reusable component architecture. Utilizing \textbf{TypeScript} ensures strict, statically typed data payloads between the asynchronous Python backend and the Javascript engine, preventing runtime type-mismatch errors commonly found in complex JSON graph endpoints.

\subsection{Cytoscape.js}
Visualizing thousands of interconnected nodes requires a robust rendering engine. \textbf{Cytoscape.js} provides an open-source, highly optimized graph theory library for visualization and analysis. It gracefully handles zooming, panning, layout calculations (Force-directed, Cola, CoSE), and DOM event interactions critical for letting researchers click through the citation trees.

\subsection{Docker Orchestration}
Containers solve the notorious "it works on my machine" software dilemma. By utilizing \textbf{Docker Compose}, the frontend webserver, the backend API server, and the Neo4j database are mathematically walled off into predictable, identical virtual environments routing through standard TCP/IP networking, significantly speeding up both development workflows and production releases.

\section{Input Design}
The system demands precise and sanitized input designs spanning two primary modes: Data Ingestion and Query Interfaces.
\begin{itemize}
    \item \textbf{Ingestion Schema}: Users provide standard JSON payloads indicating scientific keywords (e.g., `{"query": "protein folding deep learning", "sources": ["pubmed", "arxiv"]}`). The UI abstracts this behind a stylized form.

\section{Users Of The System}
The platform is designed to cater dynamically to specific subsets of the academic demographic, abstracting away backend complexity based on roles.

\subsection{Researchers}
The primary target audience. Researchers utilize the environment to instantly ingest thousands of current papers, bypass manual literature reviews, verify methodological overlaps via the Cytoscape diagram, and invoke the GraphRAG pipeline to summarize deep domain-specific knowledge accurately.

\subsection{Students}
Academic learners exploit the visual representation nature of the system to understand historical progression—visually mapping how early concepts evolved into modern methodologies, discovering foundational authors, and requesting the LLM to explain complex abstracts in simplified terms.

\subsection{System Admin / Maintainers}
Administrators manage system health, allocate Docker container resources, perform Neo4j database periodic indexing, update Ollama LLM weight models globally, and orchestrate Kubernetes scale-ups when user concurrency increases.

\section{Modularity Criteria}
The architecture heavily emphasizes the "Separation of Concerns" principle, resulting in distinct logical backend directories:
\begin{itemize}
    \item \textbf{api/}: Houses the FastAPI route controllers, strictly routing requests and maintaining HTTP state.
    \item \textbf{data\_ingestion/}: Manages asynchronous fetchers polling external sources (PubMed, arXiv API wrappers).
    \item \textbf{kg\_construction/}: Handles the conversion of raw strings. `paper_parser.py` extracts datasets/methods, while `graph_builder.py` executes Cypher inserts to Neo4j.
    \item \textbf{graph\_rag/}: The intelligence core, utilizing `query_parser.py` for intent abstraction, `graph_retriever.py` for hybrid Neo4j fetches, and `answer_generator.py` for final LLM prompt construction.
    \item \textbf{analytics/}: Generates temporal and mathematical insights based strictly on graph statistics.
\end{itemize}

Data Flow Diagrams (DFDs) trace the lifecycle and transformation of information within the software boundary.

\subsection{Basic DFD Symbols}

The system interfaces globally with **External Public Paper APIs** (External Entity) to fetch raw datasets. \begin{enumerate}

    \item \textbf{Process 2.0 (LLM Extractor)}: Translates `Raw Papers` into structured `Entities` (Methods, Authors).

\end{enumerate}

\section{Database Design}

Optimizing the schema in Neo4j differs drastically from standard SQL Normalization. In traditional SQL, 1st, 2nd, and 3rd Normal Forms attempt to minimize duplication by splitting tables. Conversely, graph schema design focuses exclusively on modeling questions the system will routinely ask. Normalization here refers strictly to creating explicit Node labels and establishing direct relationships rather than storing comma-separated property strings inside a single node.

For example, storing an array `["Author A", "Author B"]` inside a single `Paper` node breaks graph efficiency. Proper Graph Normalization extracts `Author A` into an independent `Author` node, drawing explicit `[:AUTHORED]` edges.

\section{Tables Design (Node \& Edge Specification)}

The knowledge graph utilizes the following structural models rather than standard relational tables:

\subsection{Node Entities}
\begin{itemize}
    \item \textbf{Paper}: Core entity. Contains properties: `id` (string), `title` (string), `abstract` (text), `published_date` (date), `embedding` (vector text).
    \item \textbf{Author}: Represents human contributors. Properties: `name` (string), `institution` (string, optional).
    \item \textbf{Method}: Scientific methodology mentioned. Properties: `name` (e.g., "Transformers", "CRISPR").
    \item \textbf{Dataset}: Data collection references. Properties: `name` (e.g., "ImageNet", "Kaggle COVID-19").
    \item \textbf{Institution}: Organizations or Universities.
\end{itemize}

\subsection{Edge Modalities (Relationships)}
\begin{itemize}
    \item \textbf{(Author) -[:AUTHORED\_BY]-> (Paper)}
    \item \textbf{(Paper) -[:PROPOSES\_METHOD]-> (Method)}
    \item \textbf{(Paper) -[:USES\_DATASET]-> (Dataset)}
    \item \textbf{(Paper) -[:CITES]-> (Paper)}
    \item \textbf{(Author) -[:AFFILIATED\_WITH]-> (Institution)}
\end{itemize}

\section{Backend Implementation}
The backend relies on modular Python components interacting closely.

\subsection{The Graph Builder Engine}
Located dynamically in `backend/kg_construction/graph_builder.py`, this module handles the safe, transactional ingestion of nodes ensuring data deduplication. Using Neo4j's `MERGE` clause is central to this mechanism. The Cypher query ensures duplicate nodes for the same Author are omitted, and instead, new edges are correctly drawn to existing entities. Every insertion simultaneously runs an embedding generation loop against Ollama, appending numerical vectors to the base abstract metadata.

\subsection{Retrieval-Augmented Generation Architecture}
The true innovation relies on the GraphRAG implementation found within `frontend/graph_rag/graph_retriever.py` and `answer_generator.py`. Standard vector search utilizes cosine similarity to extract the top-K papers relevant to the query string. The Graph Retriever expands this dynamically. Once the top 3 papers are identified, a secondary Cypher query executes traversing out up to 2 hops (e.g., retrieving the Methods and Co-Authors surrounding the target papers).
This topological data is subsequently mapped into a readable string and pushed to Llama 3.2 instructing the prompt:
\textit{"You are a scientific assistant. Use the following explicit graph relationships and raw abstracts to answer the query accurately..."}

\section{Frontend Visualization Implementation}
The React application (`frontend/src/App.tsx`) orchestrates the state. The complex interactions occur within the Cytoscape canvas (`frontend/src/components/NetworkGraph.tsx`). To maximize browser RAM efficiency, large network payloads over 20,000 JSON edges utilize Cytoscape's WebGL or automated layout algorithms incrementally, ensuring the user interface remains smoothly responsive at 60 FPS while physics force-directed layouts compute asynchronously. Furthermore, the aesthetics heavily employ modern 'Glassmorphism' CSS techniques—semi-transparent overlays, glowing accents, and a comprehensive dark-mode to enhance perceived product quality immensely.

The transition from a developed codebase to a reliable, maintainable production application necessitates strict operational protocols, deployment strategies, and routine maintenance schemas.

\section{System Implementation}

The architecture guarantees rapid implementation through containerization natively. Providing infrastructure as code ensures disparate hosting computers handle dependencies dynamically.

\subsection{5. Schedule Analysis}
(Note: Reflecting upon the original project planning phases, the deployment schedule aligned accurately with the predefined Agile lifecycle markers.) Moving to production involves migrating from the localized `npm run dev` and Uvicorn hot-reload servers to static frontend builds nested inside Nginx blocks mimicking true production environments.

\subsection{Implementation Methods}
Docker Compose natively maps volumes, ports, and environment variables dynamically based on a synchronized `.env` file configuration. 
Deployment executes via:
\begin{lstlisting}[language=bash, caption=Production Deployment Sequence]
$ git pull origin main
$ ollama pull llama3.2
$ docker-compose down
$ docker-compose build --no-cache
$ docker-compose up -d
\end{lstlisting}
The above completely tears down previous states (excluding explicitly mapped persistent volumes holding the Neo4j Graph Database to prevent data loss) and constructs pristine Nginx, FastAPI, and React bundles resolving instantly into operational service on Port `80`. 

Kubernetes manifests (`k8s/`) also explicitly manage rolling updates and auto-scaling implementations if institution deployment scaling dictates traffic anomalies.

\section{System Maintenance}
A robust maintenance schedule prevents systemic entropy. Routine maintenance involves:
\begin{itemize}
    \item \textbf{Data Archival and Graph Pruning}: Academic importance evolves. Periodically, older or less relevant sub-graphs can be detached or exported out of the active Neo4j visualization buffer to minimize rendering clutter while remaining archived.
    \item \textbf{Index Rebuilding}: As the graph scales iteratively past 100,000 internal nodes, triggering Cypher commands to recreate schema indices drastically restores and optimizes search times.
    \item \textbf{Model Updates}: Periodically polling Ollama registries ensuring Llama 3.2 weights incorporate modern tokenizer enhancements or performance adjustments safely.
\end{itemize}

\chapter{CONCLUSION AND FUTURE SCOPE}
\noindent{\rule{\textwidth}{1.5pt}}

\section{Conclusion}
The "Scientific Knowledge Graph Assistant" successfully bridges an incredibly complex paradigm boundary. By fusing mathematical Graph Theory topologies managed by Neo4j with the transformative generative capabilities of local Large Language Models (LLMs), this dissertation realized a zero-cost, hyper-accurate research investigation tool.

We have demonstrated conclusively that mapping scientific methodologies, dataset usages, and author networks into explicit connected graphs vastly outperforms simplistic Vector-RAG retrieval queries. The system gracefully parses textual inputs, fetches multi-hop structural context, and mandates the LLM to generate highly grounded, hallucination-free summaries while returning real-time analytics to a meticulously crafted visual user interface utilizing React and Cytoscape.js. Most significantly, this immense capability was achieved entirely locally on consumer-grade hardware guaranteeing privacy and operational zero-cost economics.

\section{Future Enhancement}
The software pipeline possesses massive extensibility potential:
\begin{itemize}
    \item \textbf{Agentic Autonomous Workflows}: Integrating secondary "Agent" LLMs capable of deciding iteratively whether to continue querying the database, traversing deeper node levels autonomously until a complex query is implicitly solved.
    \item \textbf{Real-Time PDF Ingestion Orchestration}: Utilizing local OCR algorithms (like Tesseract) to parse offline, visually scanned documents or images extracting embedded charts/graphs and merging mathematical conclusions into the nodes automatically.
    \item \textbf{Collaborative Annotations}: Allowing multiple researchers connecting via Websockets to the frontend simultaneously annotating explicitly the graph, sharing insights, and pruning unrelated scientific nodes interactively.
\end{itemize}
